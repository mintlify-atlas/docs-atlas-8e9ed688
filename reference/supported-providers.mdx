takeovflow identifies potential subdomain takeover candidates by resolving the CNAME record for each discovered subdomain and comparing it against a list of known cloud provider patterns. When a subdomain's CNAME contains a recognized pattern — for example, `github.io` or `amazonaws.com` — and the underlying cloud resource appears unclaimed, the subdomain may be vulnerable to takeover. A match does not by itself confirm a takeover; it signals that the subdomain points to a cloud service that should be investigated.

## How matching works

For each subdomain, takeovflow runs `dig <subdomain> CNAME +short` and converts the result to lowercase. The resolved CNAME string is then checked against every pattern in `CNAME_SERVICES` using a simple Python `in` membership test:

```python
if pattern in cname:
    # match — report finding with service name and severity
```

Matching is case-insensitive because the CNAME output is lowercased before comparison. The list is evaluated in order and the **first matching pattern wins** — a subdomain is reported once with the service and severity of the first match. If no pattern matches, the subdomain is not reported by the CNAME phase.

## Severity assignment

Each fingerprint carries a hardcoded severity based on how exploitable a takeover at that provider typically is:

- **HIGH** — the provider (e.g., GitHub Pages, Heroku, Netlify, Vercel) allows unauthenticated resource registration, making an unclaimed subdomain straightforwardly exploitable. Treat HIGH findings as requiring immediate manual verification.
- **MEDIUM** — the provider is a CDN or enterprise platform (e.g., CloudFront, Akamai, HubSpot, Squarespace) where a CNAME match is more likely to reflect a legitimate configuration. Findings still warrant review, but false positives are more common.

<Info>
CNAME checking runs concurrently using `ThreadPoolExecutor`, controlled by `--threads` (default: 50). On large subdomain lists this delivers up to 50x faster analysis compared to sequential `dig` calls.
</Info>
